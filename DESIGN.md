# Прога менеджера на базе GridMaster — Дизайн-документ

Документ описывает проработанный дизайн до написания кода. Всё, что зафиксировано
ниже, — результат серии решений. Открытые вопросы вынесены в отдельный раздел.

---

## 1. Бизнес-проблема (зачем всё это)

Начальник производства жалуется: менеджеры считают заказы плохо и **продают
часть заказов почти по себестоимости**, хотя клиент заплатил бы нормальную цену.
Причина — расчёт сложный, делается вручную/на глаз, маржа теряется.

**Главная ценность прогы:** не просто посчитать себестоимость (это GridMaster
уже умеет), а **подсказать менеджеру нормальную цену клиенту** и не дать продать
в минус по марже.

---

## 2. Архитектура

Выбрано: **отдельная прога менеджера + общий модуль расчёта**.

```
              ┌─────────────────────────┐
              │   Прога менеджера (UI)   │
              │  чертёж → проверка → €   │
              └────────────┬────────────┘
                           │ вызывает
                           ▼
              ┌─────────────────────────┐
              │   gridmaster_core        │  ← чистый модуль, без UI
              │   рассчитать(Заказ)→Рез  │
              └────────────┬────────────┘
                           │ тот же модуль зовёт
                           ▼
              ┌─────────────────────────┐
              │  Существующий GridMaster │
              │  (Tkinter десктоп)       │
              └─────────────────────────┘
```

Один модуль расчёта — два потребителя. Формулы не дублируются.

---

## 3. Реальное состояние кода (важно)

`Grid_Master_V50.py` — **один файл, 2568 строк, один класс `GridMasterInterface`**
(Tkinter-десктоп: вкладки, таблицы, превью раскроя на matplotlib, импорт/экспорт
Excel, проект в JSON).

Расчёт **существует** (`calculate_mat_cost`, `calculate_frame_cost`,
`calculate_angle_cost`, `calculate_bumper_cost`, `calculate_work_cost`,
`calculate_zinc_cost`, раскрой `_cut_*`), **но намертво сросся с интерфейсом**:

```python
def calculate_mat_cost(self):
    art = self.article_cb.get()              # читает виджет Tkinter
    manual = self.manual_mats_var.get()      # читает чекбокс
    L_mm, W_mm = self._get_mat_dimensions_mm()  # читает поля ввода
```

Отдельного модуля расчёта, который можно вызвать как библиотеку, **сейчас нет**.
Это определяет план: первый шаг — извлечь расчёт, а не строить UI.

## 4. Рабочий процесс менеджера

```
Новый заказ
   │
   ▼
Загрузить чертежи (PDF/DWG, можно несколько)
   │
   ▼
Claude API: чертёж → структура «Заказ» (много позиций)
   │
   ▼
Менеджер ПРОВЕРЯЕТ и подтверждает каждую позицию   ← обязательный шаг
   │  (AI не уверен → жёлтая пометка; не смог → ручной ввод)
   ▼
Расчёт через gridmaster_core
   │
   ▼
Результат: себестоимость + РЕКОМЕНДУЕМАЯ цена (ползунок маржи)
   │
   ▼
Вывод: PDF клиенту · лист списания складу · сохранить в историю
```

---

## 5. Зафиксированные решения

| Тема | Решение |
|---|---|
| Структура заказа | Один заказ = **много позиций** |
| Доверие к AI | Менеджер **обязательно проверяет** и подтверждает извлечённое |
| Вывод | Цена + лист списания складу + сохранение в историю |
| Нет на складе | Считать по **последней цене** + поле «цена поставщика» для уточнения |
| Источник цен | **Пока не решён** — модуль цен проектировать гибко (см. раздел 8) |
| История | **Локально у каждого** менеджера (не централизованно) |
| Защита маржи | **Ползунок маржи**: по умолчанию на рекомендованной (мин. +X%), менеджер видит как едет цена клиенту |
| AI не смог | **Ручной ввод как fallback** — та же форма позиции, прога всегда работает |
| Показ чертежа | PDF — рядом с формой; DWG — рендер нетривиален, возможно только ручной fallback |

## 7. План фаз

| Фаза | Что | Зачем |
|---|---|---|
| **0. Фундамент** | Извлечь расчёт из Tkinter в чистый `gridmaster_core.py` (вход — структура `Заказ`, выход — `Результат`: себестоимость + лист списания). Поведение формул **не менять**. Тесты на тождество цифр: взять реальные заказы, сверить копейка-в-копейку со старым GridMaster. | Без этого нечего переиспользовать |
| **1. Распознавание** | Claude API: чертёж PDF → структура `Заказ` (много позиций). Уверенность по полям → жёлтые пометки. | Сердце прогы менеджера |
| **2. Прога менеджера** | 4 экрана (история → приём → проверка → результат). Ручной ввод как fallback. Ползунок маржи. Вывод: PDF клиенту, лист складу, локальная история. | То, что просил шеф |
| **3. Цены и склад** | Подключить реальный источник цен и наличия (см. раздел 8). До этого — последняя известная цена + ручное поле. | Актуальность цифр |

Фазы 0–2 не блокируются вопросом источника цен. Фаза 3 — блокируется.

---

## 8. Открытые вопросы (решаются не за столом)

1. **Источник цен и наличия.** Сейчас цены живут в таблице матов внутри Tkinter
   (`self.mat_data`, грузится из конфига/Excel). Откуда прога менеджера берёт
   актуальные цены и наличие — Infotek API? выгрузка Excel? Модуль цен в Фазе 0
   проектировать как сменный интерфейс (источник за абстракцией).
2. **Показ DWG рядом с формой.** PDF реально, DWG-рендер нетривиален. Решение:
   возможно DWG идёт только через ручной fallback.
3. **Бюджет Claude API** на распознавание (страниц/чертежей в месяц).
4. **Точная формула рекомендуемой цены.** Минимальный % маржи и диапазон
   ползунка — согласовать с начальником производства (он носитель боли).

---

## 9. Принципы реализации

- Фаза 0 не меняет ни одной формулы. Только отделяет расчёт от UI. Критерий
  готовности — тесты тождества проходят на реальных заказах.
- Источник цен — за абстракцией, чтобы фаза 3 не переписывала фазу 2.
- Прога менеджера всегда работает: даже при полном отказе AI есть ручной ввод.
- Главный экран ценности — Экран 4: менеджер видит и себестоимость, и
  рекомендуемую цену клиенту.
